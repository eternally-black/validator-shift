## Product

ValidatorShift solves the validator identity migration problem that Solana operators currently solve with raw `scp`, ad-hoc bash, and crossed fingers.

**Today's options for moving a staked identity between servers:**

- `scp` the keypair: no integrity check, no audit trail, no protection against MITM, the keypair traverses the operator's transit machine in plaintext.
- Bash scripts shared as gists (e.g. STEVLTH): better than scp but no operator confirmation gates and no anti-dual-identity gating.
- Ansible playbooks: assume root SSH everywhere, full operator-side trust model — if an SSH key leaks, every validator in the playbook is compromised.

**ValidatorShift instead:**

- End-to-end encrypted with X25519 ECDH + XChaCha20-Poly1305 AEAD.
- Three-terminal SAS comparison (≈20 bits) defeats MITM at the relay.
- Per-step operator confirmation in TTY for every destructive action.
- Anti-dual-identity gate: target waits for source to stop voting before activating.
- Real rollback handlers on every step ≥ 2.
- Structured, redacted audit log persisted on the hub.
- Single-binary install (`curl | bash` with SHA-256 verification against the tagged GitHub Release).

## Target user

**Solo and small-team Solana validator operators** who need to:

- Move a staked validator to better hardware without losing stake or risking dual-signing.
- Migrate to a different cloud region or provider.
- Rotate the host underneath a long-lived identity (e.g. for OS upgrades that require host replacement rather than in-place).
- Hand off operations to a successor / colleague without exposing the keypair file in transit.

**The pain points each segment cares about:**

| Operator size | Primary concern | What ValidatorShift gives them |
|---|---|---|
| Solo validator | "I'm doing this alone at 4am, I will fumble." | Per-step confirmation prompts, copy-pasteable commands generated by the wizard, recovery runbook for every failure point. |
| Small team (2-5) | "We don't have a tooling budget." | Free, public, no vendor relationship. Hub is a coordination plane only — operators can self-host the hub if they prefer. |
| Institutional | "We need an audit trail." | Hub persists structured log events for every state transition, every operator confirmation, every preflight check; SQLite DB on a Railway named volume. |

## How to evaluate

The judging rubric maps to specific evidence in the repo:

### Execution Quality & Completeness

- Full 9-step migration state machine: [`packages/hub/src/orchestrator/state-machine.ts`](https://github.com/eternally-black/validator-shift/blob/main/packages/hub/src/orchestrator/state-machine.ts), with bilateral-step gating, valid-transition enforcement, and disconnect-resilient revival from the SQLite session row.
- Real rollback handlers for every step ≥ 2: [`packages/agent/src/commands/agent.ts`](https://github.com/eternally-black/validator-shift/blob/main/packages/agent/src/commands/agent.ts) `executeRollbackStep`. Closes the largest functional gap from the original spec.
- Tower file sanity validation before transfer: [`packages/agent/src/solana/validator.ts`](https://github.com/eternally-black/validator-shift/blob/main/packages/agent/src/solana/validator.ts) `validateTowerFile`, with [unit tests](https://github.com/eternally-black/validator-shift/blob/main/packages/agent/src/solana/validator.test.ts) covering all reject paths.
- Identity auto-derive + cross-check against running validator's `getIdentity` JSON-RPC.
- Test suite: 184 unit tests across all packages (run `npm test -ws`).

### Security

- Threat model: [`docs/THREAT_MODEL.md`](https://github.com/eternally-black/validator-shift/blob/main/docs/THREAT_MODEL.md) — STRIDE table covering all six categories with concrete mitigations.
- Crypto stack: X25519 ECDH (`@noble/curves`) → HKDF-SHA256 session key (`@noble/hashes`) → XChaCha20-Poly1305 AEAD (`@noble/ciphers`). Keys never cross the hub; the hub binary doesn't link a decryption primitive.
- SAS strength: ≈19.93 bits (6 decimal digits formatted `XXX-XXX`), exceeding the 20-bit Signal/WhatsApp safety-numbers baseline. HKDF info tag `validator-shift-sas-v2` for domain separation.
- Three-terminal SAS comparison: source TTY, target TTY, wizard card. A MITM at the hub layer presents different X25519 ephemeral keys to each agent → different SAS values → caught visually.
- Layered log redaction: [`packages/shared/src/redact.ts`](https://github.com/eternally-black/validator-shift/blob/main/packages/shared/src/redact.ts) — three independent passes (JSON-array, base64, length-based catch-all) plus a boundary sanitizer for thrown errors. 23 [adversarial tests](https://github.com/eternally-black/validator-shift/blob/main/packages/shared/src/redact.test.ts) prove keypair strings never survive in any field of an emitted log event.
- Anti-dual-identity gate: target waits for source to be observed delinquent before activating identity (`waitForSourceQuiet`). Skippable only via the explicit `--unsafe-skip-quiet-gate` flag, which is hidden behind a wizard toggle marked "Test environments only".
- Static analysis on every push: [`.github/workflows/security.yml`](https://github.com/eternally-black/validator-shift/blob/main/.github/workflows/security.yml) — `npm audit`, `osv-scanner`, `semgrep` (typescript / owasp-top-ten / nodejsscan rulesets), `actionlint`.
- Release provenance: [`.github/workflows/release.yml`](https://github.com/eternally-black/validator-shift/blob/main/.github/workflows/release.yml) builds standalone binaries with `bun --compile` from tagged commits only; install.sh verifies SHA-256 against the published `SHA256SUMS` before `chmod +x`.

### Clarity of User Experience

- Two-field wizard Step 1 (ledger + keypair). Hub URL, identity pubkey, and other plumbing values are auto-derived or build-time constants.
- Step 2 generates fully-substituted, copy-paste commands per role — no `<placeholder>` tokens left for the operator to fill.
- Cluster-type toggle on Step 1 with explicit "Test environments only" warning for the localnet escape flags. Production-mode commands are clean; localnet-mode commands include `--unsafe-skip-*` flags openly with an amber warning.
- Three-terminal SAS comparison instead of two — an extra defensive layer surfaced as a clean wizard card.
- Per-step status (running / complete / failed) reflected in the wizard StepList in real time, with a snapshot replay for late-joining dashboards.
- Recovery runbook: [`docs/RECOVERY.md`](https://github.com/eternally-black/validator-shift/blob/main/docs/RECOVERY.md) — 11-row matrix covering every failure point with copy-pasteable `agave-validator` recovery commands. Top section addresses the dual-identity risk window explicitly.

## Architecture at a glance

```
┌────────────────┐   ┌─────────────────────┐   ┌────────────────┐
│   SOURCE       │   │       HUB           │   │   TARGET       │
│   validator    │   │   (relay only)      │   │   validator    │
│                │   │                     │   │                │
│  validator-    │←──┤  WSS (encrypted) ├──→  validator-    │
│  shift agent   │   │                     │   │  shift agent   │
│                │   │  - state machine    │   │                │
│  X25519 priv ←─┼───┤  - audit log SQLite │───┼─→ X25519 priv │
│  XChaCha key   │   │  - never decrypts   │   │  XChaCha key   │
└────────────────┘   └─────────────────────┘   └────────────────┘
        ↑                      ↑                      ↑
        └──────────────  Operator (3 displays)  ──────┘
                          SAS visual compare
```

End-to-end keys are derived locally on each agent. The hub sees only ciphertext and SHA-256 hashes, plus state-machine events for the audit log.
